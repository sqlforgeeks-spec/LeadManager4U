# MULTI-SEARCH-ENGINE SUPPORT

Do not build the scraping system as a Google-only application.

Create a flexible, modular search-source architecture so that additional search engines can be added easily without redesigning the entire application.

The user should be able to choose:

## Search Engine / Platform

### Maps Sources

* Google Maps
* Bing Maps
* Apple Maps
* Other supported map/search platforms where technically feasible

### Web Search Sources

* Google Search
* Bing Search
* Yahoo Search
* DuckDuckGo
* Other search engines where technically feasible

### Image Search Sources

* Google Images
* Bing Images
* Yahoo Images
* Other image search engines where technically feasible

The exact available sources should be clearly displayed in the UI.

Do not create completely separate and duplicated scraping systems for every source.

Instead, build a common scraping framework with:

**Search Source → Query → Filters → Scraping Job → Unified Leads**

All sources should feed into the same lead database and use the same downstream workflow.

For example:

**Google Maps →**

**Bing Maps →** → **Unified Lead Database → Email Campaign**

**Google Search →**

**Bing Search →**

**Yahoo Search →**

**Google Images →**

**Bing Images →**

---

# SOURCE-SPECIFIC SCRAPING

Each search engine may provide different data.

The scraper should extract all available relevant information from each source.

Standard normalized fields should include:

* Business name
* Company name
* Category
* Email
* Phone
* Website
* Address
* City
* State/Region
* Country
* Search query
* Source platform
* Source URL
* Original result URL
* Rating where available
* Review count where available
* Description/snippet where available
* Image URL where relevant
* Date collected

Store source-specific information where useful, but normalize common fields so that all leads can be searched, filtered, exported, and used in email campaigns consistently.

---

# UNIFIED SOURCE SELECTION

The new scraping interface should make source selection obvious.

Example:

### Discover Leads

**Choose Source**

[ Google Maps ] [ Bing Maps ] [ Apple Maps ]

[ Google Search ] [ Bing Search ] [ Yahoo Search ] [ DuckDuckGo ]

[ Google Images ] [ Bing Images ] [ Yahoo Images ]

The interface should dynamically display only the filters relevant to the selected source.

Do not force users to see dozens of irrelevant settings.

---

# SOURCE FILTERS

Create a common filter system wherever technically applicable:

* Search phrase/query
* Location
* Country
* City
* Language
* Result limit
* Data required:

  * Email
  * Phone
  * Website
* Category
* Date collected
* Duplicate handling

For example, users should be able to say:

**Find:**
"Marketing agencies"

**Location:**
"New York"

**Source:**
Google Search + Bing Search + Yahoo Search

**Required data:**
Email + Website

Then start one scraping job that collects and combines results from the selected sources.

---

# MULTI-SOURCE JOBS

Allow users to select one or multiple sources for a scraping job.

Example:

**Query:** Real estate agencies

**Locations:** New York, Chicago, Los Angeles

**Sources:**
✓ Google Maps
✓ Bing Maps
✓ Google Search
✓ Bing Search
✓ Yahoo Search

**Required fields:**
✓ Email
✓ Phone
✓ Website

**Start Scraping**

The system should then show the source of every result and prevent duplicate leads where possible.

---

# DEDUPLICATION

Because the same business may appear across multiple search engines, implement intelligent deduplication.

Use available matching signals such as:

* Normalized business name
* Website domain
* Email address
* Phone number
* Address
* Location

The same business found on multiple sources should ideally become one unified lead with multiple source references.

Example:

**ABC Marketing Agency**

Found on:

* Google Maps
* Google Search
* Bing Search
* Yahoo Search

The lead should not appear four times unnecessarily.

Instead, preserve the source information inside the unified lead record.

---

# SOURCE PERFORMANCE

For every scraping job, show useful source-level information:

* Leads collected by source
* Unique leads
* Duplicate leads
* Emails found
* Phones found
* Websites found
* Errors/warnings by source

Example:

Google Maps:

* 1,200 results
* 800 unique leads
* 420 emails

Google Search:

* 900 results
* 500 unique leads
* 310 emails

Yahoo Search:

* 700 results
* 280 unique leads
* 150 emails

This allows users to understand which sources are producing the best leads.

---

# MODULAR ARCHITECTURE

Architect the scraping system so each source is a modular provider/adapter.

For example, conceptually:

* GoogleMapsProvider
* BingMapsProvider
* AppleMapsProvider
* GoogleSearchProvider
* BingSearchProvider
* YahooSearchProvider
* DuckDuckGoProvider
* GoogleImagesProvider
* BingImagesProvider

All providers should follow a common interface and return normalized lead data.

This makes it easy to add new sources in the future without rewriting the entire application.

The UI should also make adding future search providers straightforward.

---

# IMPORTANT

Before implementing each source, verify what data can actually be collected reliably and design the scraper around the source's actual structure.

Do not assume that every search engine exposes the same data.

The application should gracefully handle:

* Missing emails
* Missing phone numbers
* Missing websites
* Different result structures
* Source-specific errors
* Rate limits
* Temporary failures
* Partial scraping completion

A failure in one source should not unnecessarily stop the entire multi-source scraping job.

The final system should be capable of becoming a multi-source lead intelligence platform rather than remaining limited to a single search engine.
